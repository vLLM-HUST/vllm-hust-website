🚀 Quick Start (v0.5)

Three commands to first token.

📦 1. Pip Install
$ pip install isagellm
🔧 2. Setup Backend
# Install CUDA, Ascend, or CPU backend automatically $ sagellm install cuda # or ascend / cpu
⚡ 3. Run Inference
$ sagellm run -p "Hello AI" --backend cuda
💡 Smart Detection: v0.5 automatically detects if you're missing dependencies for your hardware and suggests the correct sagellm install command.

🏗️ System Architecture

Layered design with protocol-first approach, CPU-first development, and PD separation for optimal performance.

Current Stack
Unified entry, routed execution, layered runtime, protocol-first foundation.
Entry Layer

Clients, CLI, Benchmark, and Gateway

Users can enter through the unified sagellm CLI, OpenAI-compatible REST gateway, SDKs, or benchmark clients.

Unified Entry sagellm run, sagellm chat, sagellm serve
Open Interfaces OpenAI-compatible HTTP API, streaming responses, benchmark endpoints
↓
Control Layer

Control Plane for Routing, Scheduling, and Lifecycle

The control plane selects engines, applies scheduling policy, tracks health, and keeps multi-engine execution manageable.

Routing RequestRouter, engine selection, fail-fast health checks
Scheduling FIFO, priority, SLO-aware policy hooks, lifecycle management
↓
Execution Layer

Core Engine Runtime

The core runtime owns model execution, worker orchestration, PD separation, metrics, and the internal execution path for native SageLLM inference.

Runtime Units LLMEngine, ModelRunner, Worker, runtime plugins, metrics collection
Inference Semantics Prefill/decode split, batch execution, KV-aware generation path
↓
Backend CPU, CUDA, Ascend device abstraction, memory, stream, and kernel boundary.
Comm Distributed transport primitives, topology, collectives, and point-to-point communication.
KV Cache Allocation, eviction, migration, and prefix-aware cache management across inference steps.
Compression Quantization, sparsity, speculative and acceleration-oriented algorithm modules.
↓
Protocol Foundation

Shared Types, Errors, Metrics, and IDs

Protocol-first governance keeps request schemas, error codes, metrics fields, and tracing identifiers consistent across the stack.

Request Flow Client → Gateway or CLI → Control Plane → Core Engine → Backend / KV / Comm.
Design Rule Lower layers never depend on higher layers; third-party compare logic stays in benchmark, not core runtime.
Operational Focus CPU-first defaults, Ascend/CUDA acceleration, observability, and fail-fast behavior.

✅ PD Separation

Prefill and Decode phases run independently for optimal resource utilization

✅ CPU-First

All components work on CPU by default, GPU/NPU are optional accelerators

✅ Observable

Full metrics tracing with trace_id, request_id, timestamps at every layer
